# Security Policy

## Public demo boundary

This repository is a public demo-build surface. It may contain public-safe build artifacts, demo manifests, sample data, receipts, website bridge code, and documentation.

It must not contain private credentials, production keys, customer data, payment data, internal source code, family-office records, or private operational dashboards.

## Supported surface

Security reports for the public Dominion OS demo-build surface should focus on public asset safety, private-data exposure risk, stale public claims, unsafe external links, unapproved runtime origins, browser security regressions in the `/demo-1` bridge package, dependency health, workflow health, and release-artifact integrity.

## Reporting

Report suspected security issues privately through the Fractal5 Solutions website contact path.

Do not place private credentials, customer data, payment data, or non-public operational details into public issues or discussions.

## Claim-control rules

The public demo may claim a live controlled demo only when the relevant routes and assets are verified.

The public demo must not claim full commercial production readiness, live commerce readiness, customer-portal readiness, native self-healing, direct MP4 availability, or downloadable binary availability without separate current receipts.

## Current posture

Security posture is public-demo hardened, not full commercial green, until secret-scanning receipts, CI health, observability receipts, commerce proof, and customer-provisioning evidence are all current.
